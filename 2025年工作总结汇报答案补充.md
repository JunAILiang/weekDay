### 1.1 PWA项目0到1（5月上线）

**Q: 为什么能在5月上线？技术选型依据是什么？**

**答案**：
```

【技术选型理由】
1. PWA vs Native App:
   - 摆脱应用商店限制,解决FB广告投放落地页问题
   - 开发周期短,1个月即可上线(Native App需3-6个月)
   - 无需审核,可快速迭代

2. 技术栈选择:
   - 基于现有H5版本改造,复用70%代码
   - SSR架构保证首屏性能
   - Service Worker实现离线能力

【开发周期控制】
- 第1-2周:架构设计+账户体系搭建
- 第3周:核心功能开发(匹配、呼叫、支付)
- 第4周:测试+优化+上线

【核心价值验证】
- 上线后iOS用户可直接安装,不依赖App Store
- 建立独立分发能力,为后续投放放量奠定基础
```

```

1. 安装能力
   - PWA:可安装到桌面,独立App图标,全屏运行
   - H5:浏览器内运行,无法安装

2. 用户体验

   - H5:浏览器地址栏常驻,影响沉浸感

3. 投放价值
   - PWA:FB广告可直投PWA落地页,安装转化率更高
   - H5:只能引导到网页,转化路径长
```

### 1.2 性能突破（启动时间↓60.6%、包体积↓39.7%）

**Q: 具体做了哪些优化？为什么能达到这个幅度？**

**答案**：
```
性能优化的4个核心技术手段:

【1. SSR优化】
- 优化前:客户端渲染,首屏白屏2-3秒
- 优化后:服务端渲染,首屏直出HTML,白屏时间<0.5秒
- 收益:首屏加载时间从3秒→1.2秒(↓60%)

【2. 代码分割】
- 优化前:单个bundle 2.5MB,加载慢
- 优化后:按路由分割,首页bundle仅800KB
- 收益:包体积从2.5MB→1.5MB(↓40%)

【3. 图片优化】
- 主播详情页图片秒展示(10月14日上线)
- 懒加载+预加载策略
- WebP格式替代PNG/JPG,体积减少30%

【4. 资源压缩】
- Gzip压缩,传输体积减少70%
- 图片压缩,清晰度不变

【数据验证】
- 启动时间:从3.0秒→1.18秒(↓60.6%)
- 包体积:从2.5MB→1.51MB(↓39.7%)
- 用户首启流失率:从35%→22%(↓37%)
```
