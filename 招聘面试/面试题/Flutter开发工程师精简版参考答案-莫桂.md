# Flutter开发工程师精简版参考答案-莫桂

候选人：莫桂  
岗位方向：Flutter开发工程师（偏 Android / Flutter 架构）  
经验年限：多年移动端研发经验

## 说明

这份内容与《Flutter开发工程师精简版面试题-莫桂》一一对应，适合现场快速判断。
莫桂这类候选人不能只按普通 Flutter 开发来问，重点是验证：
1. 架构和工程治理是不是他真做过；
2. Flutter 从 0 到 1 的落地深度；
3. Android 原生、混合架构、模块化治理是否扎实；
4. AI 赋能研发是否有真实实践，而不是简历包装。

## 1. Android 原生转向 Flutter 的关键节点

好的回答应包含：
- 能讲清从 Android 原生到 Flutter 的演进背景；
- 不是“因为流行所以学”，而是基于业务、团队、跨端复用效率做判断；
- 能说明自己原生基础如何帮助 Flutter 落地。

较弱回答表现：
- 只说 Flutter 开发效率高；
- 说不清什么时候真正开始做 Flutter；
- 原生与 Flutter 的关系讲不透。

建议追问：
- 你原生背景里哪些能力最直接迁移到了 Flutter？
- 如果团队只有你支持 Flutter，你会先做什么？

## 2. Flutter 技术栈从 0 到 1 落地

好的回答应包含：
- 能说清前期调研范围、对比过哪些方案、最终如何选型；
- 有明确的架构、状态管理、目录结构、规范输出；
- 不只是做了一个项目，而是推动团队形成可复制的 Flutter 开发方式。

较弱回答表现：
- 把“0 到 1”说成“做了第一个 Flutter 页面”；
- 没有规范、没有方法论、没有技术决策细节；
- 说不清团队为什么接受这套方案。

建议追问：
- 你当时对比了哪些状态管理方案？为什么最终这么定？
- 团队在引入 Flutter 时最大的阻力是什么？

## 3. Bitsky 混合架构完整链路

好的回答应包含：
- 能把 Native 容器、小程序平台、Flutter 页面、PlatformChannel、JS SDK 注入之间的关系讲清；
- 知道同步/异步通信、事件回调、错误处理如何设计；
- 对 add-to-app 场景、路由栈、后台服务监听有真实理解。

较弱回答表现：
- 只会重复简历里的术语；
- 说不清链路和边界；
- 一追问实现细节就空。

建议追问：
- 这套通信链路里最容易出问题的地方是什么？
- 你们怎么保证 Native 和 Flutter 之间状态一致？

## 4. 为什么一定要做 engine 复用

好的回答应包含：
- 知道多个 engine 会带来额外的内存、启动和上下文成本；
- 能讲清 engine 生命周期管理、资源复用、页面切换体验；
- 能结合自己项目讲为什么复用是必要的。

较弱回答表现：
- 只会说“为了性能更好”；
- 说不清不复用到底会怎样；
- 对 engine 机制理解浅。

建议追问：
- 你们怎么管理 engine 的初始化和销毁？
- engine 复用有没有带来新问题？

## 5. 丰云行里的 Android 架构转型

好的回答应包含：
- 能讲清从 MVP 到 MVVM、androidx 迁移、Kotlin First 的真实背景；
- 知道为什么旧架构扛不住业务增长；
- 能讲出迁移中最大的技术和组织阻力。

较弱回答表现：
- 只会说“为了更先进”；
- 没有迁移策略；
- 说不清如何控制风险。

建议追问：
- 你们是渐进式迁移还是一次性切？
- 当时团队最大的分歧点是什么？

## 6. 多工程多 module 治理

好的回答应包含：
- 能讲清为什么原有模式限制了协作效率和构建效率；
- 知道模块边界、依赖管理、独立编译、接口规范的关键作用；
- 能给出实施后的具体收益。

较弱回答表现：
- 只会说模块化好；
- 说不清具体拆法和收益；
- 工程治理经验停留在概念层。

建议追问：
- 你们怎么定义 module 间依赖边界？
- 多工程后最大的管理成本是什么？

## 7. 长图滚动优化与 MIDI 监听抽象

好的回答应包含：
- 长图优化能讲到图片加载、内存占用、滚动流畅度之间的平衡；
- MIDI 抽象能讲到为什么原来逻辑分散、如何统一监听管理、收益是什么；
- 两个案例都应体现“真实解决问题”，而不是概念复述。

较弱回答表现：
- 只能讲结果，讲不出过程；
- 说不清做了哪些具体优化；
- 抽象能力和工程能力体现不出来。

建议追问：
- 长图优化里，你怎么判断是图片问题而不是布局问题？
- MIDI 统一管理后，最明显减少了哪类问题？

## 8. websocket 双向实时通信与课堂状态同步

好的回答应包含：
- 能讲清老师端与学生端的状态同步模型；
- 知道消息顺序、断线重连、重复消息、状态一致性等典型问题；
- 能体现客户端在实时场景中的责任边界。

较弱回答表现：
- 只会说“用了 websocket”；
- 讲不清同步机制；
- 缺乏实时系统基本理解。

建议追问：
- 如果教师端和学生端状态不同步，你先查哪一层？
- 弱网下你们做了哪些兜底？

## 9. 小票打印架构设计

好的回答应包含：
- 知道为什么要做模版层、引擎层、驱动层分层；
- 能讲清如何兼容不同设备、不同指令、不同模版；
- 能说明这套设计对复用和扩展的价值。

较弱回答表现：
- 只说“做了打印组件”；
- 说不清架构分层；
- 抽象能力不足。

建议追问：
- 你们怎么做到设备无关？
- 这套架构上线后最大的收益是什么？

## 10. 最能证明 Flutter 底层理解的案例

好的回答应包含：
- 至少能结合渲染、状态更新、原生通信、性能调优中的一个真实案例展开；
- 不停留在组件 API 使用层；
- 能体现自己对机制而不是表象的理解。

较弱回答表现：
- 只会背 Flutter 原理；
- 讲不出真实项目中的对应案例；
- 理论和实践脱节。

建议追问：
- 你遇到过最典型的 Flutter 性能问题是什么？
- 你怎么判断一个问题该去查 Flutter 层还是 Native 层？

## 11. Coding AI 在研发中的使用边界

好的回答应包含：
- 能具体说出 AI 用在代码生成、测试、调试、文档、review 建议中的哪些环节；
- 有质量控制机制，不会盲信 AI 输出；
- 知道 AI 擅长什么，不擅长什么。

较弱回答表现：
- 只说提升效率；
- 说不清如何验收 AI 结果；
- 把 AI 说成万能工具。

建议追问：
- 哪类代码你绝不会直接接受 AI 输出？
- AI 在架构设计上帮你最大的地方是什么？

## 12. 新的 Flutter + Native 混合项目怎么定基线

好的回答应包含：
- 能从架构边界、模块拆分、通信机制、工程规范、日志监控、发布流程等维度完整回答；
- 有先后顺序和取舍意识；
- 体现出能承担主程或架构职责。

较弱回答表现：
- 回答零散，没有体系；
- 只谈理想方案，不谈交付约束；
- 缺少可落地性。

建议追问：
- 你第一周最先会定哪三件事？
- 哪些东西你会先不做，等业务稳定再补？

## 快速判断建议

如果莫桂能把混合架构、Flutter 0 到 1、Android 架构治理、实时通信、小票打印这几块都讲到“背景 + 取舍 + 实现 + 结果”，那基本说明是偏资深、偏架构型候选人。
如果很多内容只是简历术语堆砌，一追问细节就空，尤其在 engine 复用、模块治理、实时通信和 AI 工程落地上发虚，就要警惕包装成分。
