# 发包全流程手册

> 适用场景：iOS 马甲包从账号准备到封号处理的完整链路
> 创建时间：2026年3月9日
> 关联文档：[iOS马甲包制作流程图.md](iOS马甲包制作流程图.md)

---

## 一、前期准备

### 1.1 账号购买标准

| 维度 | 要求 |
|------|------|
| 供应商优先级 | 美国新供应商优先，禁用养号历史列表 |
| 账号类型 | 未使用过的新号，避免有提审历史的账号 |
| 国家分配 | 新 App 账号分散不同国家，不集中在同一地区 |
| 养号策略 | 当前结论：养号效果不明显，反而影响迭代节奏，暂停养号 |

### 1.2 IP 购买规范

| 维度 | 要求 |
|------|------|
| 类型 | 住宅 IP（Residential），禁用机房 IP |
| 地区 | 优先欧洲，购买时勾选质量选项 |
| 隔离原则 | 一个 IP 只绑定一个账号，严禁复用 |
| 污染判断 | 该 IP 曾出现卡审/封号记录，视为污染，停用 |

### 1.3 账号管理表维护

每个账号需记录以下字段，统一维护在账号管理表中：

- IP 地址
- 用户名 / 密码
- 过期时间
- 当前状态（可用 / 验号中 / 卡审 / 封号）
- 绑定包名
- 备注（异常记录）

## 二、验号 SOP

> 目标：确认账号可正常登录、网络环境干净、资料完整，为发包做好准备
> 参考耗时：20 分钟/个

### 2.1 验号 Checklist

- [ ] 切换至对应住宅 IP，确认 IP 归属地正确
- [ ] 登录 Apple Developer 账号，确认可正常访问
- [ ] 修改账号密码（首次使用必须修改）
- [ ] 确认账号未被锁定、未有异常提示
- [ ] 检查账号是否已有 App 记录（新号应为空）
- [ ] 确认双重认证设备绑定正常
- [ ] 填写/更新账号管理表，状态改为"可用"

### 2.2 验号结果判断

| 结果 | 判断标准 | 处理方式 |
|------|---------|---------|
| 通过 | 登录正常、IP 干净、资料完整 | 进入开发循环 |
| 异常 | 登录失败、账号锁定、IP 被识别为异常 | 更换账号或 IP，重新验号 |
| 卡审风险 | 账号有历史提审记录且曾卡审 | 降级使用或弃用 |

## 三、发包 SOP

> 参考耗时：提审流程本身约 1 小时；A1 开发 1~1.5 天为最大耗时环节

### 3.1 发包前验收 Checklist

**A 面（A1 + A2）**
- [ ] A1 UI 还原度达标，无明显抄袭特征
- [ ] A2 内购功能配置正确，内购路径验证通过
- [ ] 代码混淆已执行
- [ ] 第三方库已替换/优化（避免代码重复问题）

**资产**
- [ ] Logo 已申请并到位
- [ ] 五图（截图）已准备，内容与 App 功能匹配
- [ ] 应用名称已确认，无重复

**提审资料**
- [ ] 银行卡已申请绑定
- [ ] App 描述填写完整，语言与目标市场匹配
- [ ] 隐私政策链接有效
- [ ] 年龄分级填写正确

### 3.2 并行发包策略

- 单账号串行等待审核期间，可同步推进其他账号的开发
- 当前优先顺序：A1 → A2 → H5B，不等 A1 过审再开始 A2
- 卡审账号（等待超过 3 天）：撤审后更换账号重提，不在原账号上继续等待

## 四、拒审处理

> 拒审后需在 24 小时内完成原因分析，48 小时内给出修复方案

### 4.1 拒审原因分类

| 类型 | 典型表现 | 优先级 |
|------|---------|--------|
| 4.3 设计相似 | 与已有 App 图标/界面高度相似 | 高 |
| 代码重复 | 使用了与其他 App 相同的第三方库或代码结构 | 高 |
| 内购问题 | 内购路径不完整、定价异常 | 高 |
| 内容违规 | 描述/截图含违规内容 | 中 |
| 资料不完整 | 隐私政策、年龄分级等缺失 | 低 |

### 4.2 各类原因修复方案

**4.3 设计相似**
- 调整 Logo（换色系、换图形元素）
- 修改主界面 UI 布局，增加差异化设计
- 更换应用名称

**代码重复**
- 替换或封装第三方库
- 优化核心功能模块，增加差异化逻辑
- 代码混淆加强

**内购问题**
- 验证内购主路径完整性（参考 Refiy 案例：内购未跑到正确版位）
- 重新配置 App Store Connect 内购项目

### 4.3 拒审记录模板

每次拒审需记录：

```
- 包名：
- 拒审时间：
- 拒审原因（Apple 原文）：
- 原因分类：
- 修复方案：
- 重提时间：
- 重提结果：
```

## 五、封号处理

> 封号是损失已发生的情况，核心动作是止损 + 根因分析 + 补充机制

### 5.1 封号后即时动作（1 小时内）

- [ ] 确认封号类型（账号封禁 / 4.3 下架 / 违规下架）
- [ ] 评估该账号下有多少包在投放，通知投放侧及时切换
- [ ] 将账号状态更新为"封号"，记录封号时间和原因
- [ ] 检查同批次账号（同供应商、同 IP 段）是否有连带风险

### 5.2 根因分析模板（24 小时内完成）

```
- 封号账号：
- 封号时间：
- 封号原因（已知/推测）：
- 该账号 IP 情况：
- 该账号发包历史：
- 是否有关联账号受影响：
- 根因判断：□ 账号质量问题  □ IP 污染  □ 内容违规  □ 其他
- 改进措施：
```

### 5.3 账号池补充机制

| 触发条件 | 动作 |
|---------|------|
| 可用账号 < 5 个 | 立即补充采购，优先美国新供应商 |
| 连续封号 ≥ 3 个 | 暂停当前供应商，排查 IP 和账号质量问题 |
| 卡审率 > 30% | 全面审查账号来源和 IP 配置 |

## 六、数据看板（周维度）

每周周报需包含以下数据，主管重点关注异常波动：

| 指标 | 说明 | 健康基准 |
|------|------|---------|
| 提审数量 | A面 + B面分开统计 | - |
| 过审数量 / 过审率 | 过审数 ÷ 提审数 | ≥ 80% |
| 卡审数量 | 超过 3 天未出结果 | 越少越好 |
| 拒审数量 / 拒审原因分布 | 按类型统计 | - |
| 封号数量 | 含 4.3 下架 | 0 为目标 |
| 可用账号库存 | 当前可用账号数 | ≥ 10 个 |

## 七、主管参与节奏建议

| 阶段 | 参与方式 | 目标 |
|------|---------|------|
| 第 1-2 周 | 亲自全程参与每个环节 | 建立第一手认知，输出各环节 checklist |
| 第 3-4 周 | 抽查执行，重点跟进拒审和封号 | 验证 SOP 有效性，持续完善 |
| 第 5 周起 | 看数据看板，异常时介入 | 转为监控模式，团队独立执行 |
